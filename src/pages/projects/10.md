<h2 data-locale-en="About this project" data-locale-es="Sobre este proyecto">About this project</h2>

<p data-locale-en="Alum-net was my final degree project for the Tecnologo en Informatica program at FING. The assignment was to build an initial virtual learning environment for an educational institution, with a Java backend, a web client for all roles, and an Android mobile app for students." data-locale-es="Alum-net fue mi proyecto de tesis para recibirme de Tecnologo en Informatica en la FING. La consigna consistia en construir una version inicial de un entorno virtual de aprendizaje para una institucion educativa, con backend en Java, cliente web para todos los roles y una app mobile Android para estudiantes.">Alum-net was my final degree project for the Tecnologo en Informatica program at FING. The assignment was to build an initial virtual learning environment for an educational institution, with a Java backend, a web client for all roles, and an Android mobile app for students.</p>

<p data-locale-en="Beyond the functional scope, the project also emphasized technology research and architectural decisions. That meant the work was not only about implementing features, but also about evaluating tools, validating them through proof-of-concept work, and understanding how they behaved in a full system." data-locale-es="Mas alla del alcance funcional, el proyecto tambien ponia enfasis en la investigacion de tecnologias y en las decisiones de arquitectura. Eso hizo que el trabajo no fuera solo implementar funcionalidades, sino tambien evaluar herramientas, validarlas mediante pruebas de concepto y entender como se comportaban dentro de un sistema completo.">Beyond the functional scope, the project also emphasized technology research and architectural decisions. That meant the work was not only about implementing features, but also about evaluating tools, validating them through proof-of-concept work, and understanding how they behaved in a full system.</p>

<h2 data-locale-en="What we built" data-locale-es="Que construimos">What we built</h2>

<p data-locale-en="The platform included three user roles: administrator, teacher, and student. Administrators could create users and courses, both individually and in bulk. Teachers could organize course content, upload materials, manage enrollments, create tasks and questionnaires, publish grades, and communicate with students. Students could browse their courses, download materials, submit assignments, participate in forums, send messages, and review their grades from both web and mobile." data-locale-es="La plataforma incluia tres roles de usuario: administrador, docente y estudiante. Los administradores podian crear usuarios y cursos, tanto de forma individual como masiva. Los docentes podian organizar contenidos, subir materiales, gestionar matriculaciones, crear tareas y cuestionarios, publicar calificaciones y comunicarse con estudiantes. Los estudiantes podian navegar sus cursos, descargar materiales, entregar tareas, participar en foros, enviar mensajes y consultar sus notas desde web y mobile.">The platform included three user roles: administrator, teacher, and student. Administrators could create users and courses, both individually and in bulk. Teachers could organize course content, upload materials, manage enrollments, create tasks and questionnaires, publish grades, and communicate with students. Students could browse their courses, download materials, submit assignments, participate in forums, send messages, and review their grades from both web and mobile.</p>

<p data-locale-en="This was not conceived as a simple document repository. The goal was to build a coherent academic platform with role-based access, communication tools, course management, evaluation flows, and a user experience adapted to both desktop and mobile contexts." data-locale-es="No estaba pensado como un simple repositorio de archivos. La meta era construir una plataforma academica coherente, con control de acceso por roles, herramientas de comunicacion, gestion de cursos, flujos de evaluacion y una experiencia adaptada tanto a escritorio como a mobile.">This was not conceived as a simple document repository. The goal was to build a coherent academic platform with role-based access, communication tools, course management, evaluation flows, and a user experience adapted to both desktop and mobile contexts.</p>

<h2 data-locale-en="Technical decisions" data-locale-es="Decisiones tecnicas">Technical decisions</h2>

<p data-locale-en="The backend was built with Java and Spring Boot, following a structure that separated domain logic, application services, and infrastructure concerns. For persistence, we used PostgreSQL for the transactional part of the platform and MongoDB for the messaging module, which had different data access patterns and real-time requirements." data-locale-es="El backend fue desarrollado con Java y Spring Boot, siguiendo una estructura que separaba logica de dominio, servicios de aplicacion e infraestructura. Para persistencia usamos PostgreSQL en la parte transaccional de la plataforma y MongoDB para el modulo de mensajeria, que tenia patrones de acceso y necesidades de tiempo real diferentes.">The backend was built with Java and Spring Boot, following a structure that separated domain logic, application services, and infrastructure concerns. For persistence, we used PostgreSQL for the transactional part of the platform and MongoDB for the messaging module, which had different data access patterns and real-time requirements.</p>

<p data-locale-en="On the frontend side, we chose React Native with Expo to share a significant part of the codebase between the web app and the Android application. That decision helped us keep business flows more consistent across platforms while reducing duplication. We also integrated Keycloak for identity and access control, WebSockets for private messaging, AWS S3 for file storage, and CI/CD workflows with GitHub Actions and Docker-based environments." data-locale-es="Del lado del frontend elegimos React Native con Expo para compartir una parte importante de la base de codigo entre la aplicacion web y la app Android. Esa decision nos ayudo a mantener flujos de negocio mas consistentes entre plataformas y a reducir duplicacion. Tambien integramos Keycloak para identidad y control de acceso, WebSockets para mensajeria privada, AWS S3 para almacenamiento de archivos y workflows de CI/CD con GitHub Actions y entornos basados en Docker.">On the frontend side, we chose React Native with Expo to share a significant part of the codebase between the web app and the Android application. That decision helped us keep business flows more consistent across platforms while reducing duplication. We also integrated Keycloak for identity and access control, WebSockets for private messaging, AWS S3 for file storage, and CI/CD workflows with GitHub Actions and Docker-based environments.</p>

<h2 data-locale-en="Challenges and learning" data-locale-es="Desafios y aprendizajes">Challenges and learning</h2>

<p data-locale-en="One of the hardest parts of the project was the integration work. Mapping Keycloak roles into Spring Security authorities, validating JWT tokens correctly, and coordinating changes between backend and frontend required much more attention than the feature list alone suggested. At several points, internal backend refactors changed API response shapes and forced us to realign the clients and the automated tests." data-locale-es="Una de las partes mas dificiles del proyecto fue el trabajo de integracion. Mapear roles de Keycloak hacia autoridades de Spring Security, validar correctamente los JWT y coordinar cambios entre backend y frontend requirio mucha mas atencion de la que sugeria la lista de funcionalidades. En varios momentos, refactors internos del backend cambiaron estructuras de respuesta de la API y nos obligaron a realinear clientes y pruebas automatizadas.">One of the hardest parts of the project was the integration work. Mapping Keycloak roles into Spring Security authorities, validating JWT tokens correctly, and coordinating changes between backend and frontend required much more attention than the feature list alone suggested. At several points, internal backend refactors changed API response shapes and forced us to realign the clients and the automated tests.</p>

<p data-locale-en="Deployment also pushed the project into more realistic territory. Keycloak, certificates, reverse proxy configuration with Nginx, and automated deployments to AWS made infrastructure an active part of the development process. At the same time, the original scope was more ambitious than what could be implemented comfortably, so we had to prioritize a solid MVP over a broader but weaker delivery. That tradeoff became one of the most useful lessons of the entire project." data-locale-es="El despliegue tambien llevo el proyecto a un terreno mucho mas realista. Keycloak, certificados, configuracion de reverse proxy con Nginx y despliegues automatizados en AWS hicieron que la infraestructura fuera una parte activa del desarrollo. Al mismo tiempo, el alcance original era mas ambicioso de lo que podia implementarse con comodidad, asi que tuvimos que priorizar un MVP solido por encima de una entrega mas amplia pero mas fragil. Ese equilibrio termino siendo uno de los aprendizajes mas valiosos de todo el proyecto.">Deployment also pushed the project into more realistic territory. Keycloak, certificates, reverse proxy configuration with Nginx, and automated deployments to AWS made infrastructure an active part of the development process. At the same time, the original scope was more ambitious than what could be implemented comfortably, so we had to prioritize a solid MVP over a broader but weaker delivery. That tradeoff became one of the most useful lessons of the entire project.</p>
