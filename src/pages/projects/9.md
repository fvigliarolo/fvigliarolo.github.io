<h2 data-locale-en="About this project" data-locale-es="Sobre este proyecto">About this project</h2>

<p data-locale-en="ServiPuntosUy is a multi-tenant loyalty platform designed for gas station chains." data-locale-es="ServiPuntosUy es una plataforma multi-tenant de fidelización pensada para cadenas de estaciones de servicio.">ServiPuntosUy is a multi-tenant loyalty platform designed for gas station chains.</p>

<p data-locale-en="The system allows each tenant to manage its own loyalty configuration, promotions, branches, users, and visual identity, while sharing the same technological base. The project includes a .NET backend, an administrative backoffice built with .NET MVC, an end-user web application in React, and a mobile application built with .NET MAUI." data-locale-es="El sistema permite que cada tenant gestione su propia configuración de fidelización, promociones, sucursales, usuarios e identidad visual, compartiendo la misma base tecnológica. El proyecto incluye un backend en .NET, un backoffice administrativo construido con .NET MVC, una aplicación web para usuarios finales en React y una aplicación mobile hecha con .NET MAUI.">The system allows each tenant to manage its own loyalty configuration, promotions, branches, users, and visual identity, while sharing the same technological base. The project includes a .NET backend, an administrative backoffice built with .NET MVC, an end-user web application in React, and a mobile application built with .NET MAUI.</p>

<p data-locale-en="The platform was created as part of a college project, but the scope pushed us to think beyond a basic CRUD application. We had to design a system with different user roles, tenant-aware behavior, QR-based redemption flows, and support for features such as notifications and age verification through a mock external service." data-locale-es="La plataforma surgió como parte de un proyecto de facultad, pero el alcance nos empujó a pensar mucho más allá de una aplicación CRUD básica. Tuvimos que diseñar un sistema con distintos roles de usuario, comportamiento consciente del tenant, flujos de canje con QR y soporte para funcionalidades como notificaciones y verificación de edad a través de un servicio externo simulado.">The platform was created as part of a college project, but the scope pushed us to think beyond a basic CRUD application. We had to design a system with different user roles, tenant-aware behavior, QR-based redemption flows, and support for features such as notifications and age verification through a mock external service.</p>

<h2 data-locale-en="Walkthrough" data-locale-es="Recorrido">Walkthrough</h2>

<p data-locale-en="One of the most important challenges in this project was the multi-tenant architecture." data-locale-es="Uno de los desafíos más importantes de este proyecto fue la arquitectura multi-tenant.">One of the most important challenges in this project was the multi-tenant architecture.</p>

<p data-locale-en="Instead of creating isolated deployments for each client, we implemented a shared platform where tenant resolution depends on the request context. For administrators, tenant and role are identified through JWT claims after login. For end users, the system can resolve the tenant from the subdomain, and for mobile scenarios we also supported tenant resolution through headers. This allowed the same backend to serve different experiences depending on who was making the request." data-locale-es="En lugar de crear despliegues aislados para cada cliente, implementamos una plataforma compartida donde la resolución del tenant depende del contexto de la request. Para administradores, tenant y rol se identifican a través de claims JWT después del login. Para usuarios finales, el sistema puede resolver el tenant desde el subdominio y, en escenarios mobile, también soportamos resolución por headers. Esto permitió que el mismo backend atendiera experiencias distintas según quien estuviera haciendo la request.">Instead of creating isolated deployments for each client, we implemented a shared platform where tenant resolution depends on the request context. For administrators, tenant and role are identified through JWT claims after login. For end users, the system can resolve the tenant from the subdomain, and for mobile scenarios we also supported tenant resolution through headers. This allowed the same backend to serve different experiences depending on who was making the request.</p>

<p data-locale-en="From an architectural perspective, we worked with a layered structure and a service organization that separated behavior by user type. This made it easier to model the responsibilities of platform administrators, tenant administrators, branch administrators, and end users without collapsing all the logic into the same service layer." data-locale-es="Desde el punto de vista arquitectonico, trabajamos con una estructura en capas y una organizacion de servicios que separaba comportamientos segun el tipo de usuario. Eso facilito modelar las responsabilidades de administradores de plataforma, administradores de tenant, administradores de sucursal y usuarios finales sin mezclar toda la logica en una misma capa de servicios.">From an architectural perspective, we worked with a layered structure and a service organization that separated behavior by user type. This made it easier to model the responsibilities of platform administrators, tenant administrators, branch administrators, and end users without collapsing all the logic into the same service layer.</p>

<p data-locale-en="Another interesting part of the project was the loyalty flow itself. The platform supports points accumulation, product and promotion management, branch-specific operations, transaction history, and QR-based redemption. That last feature was especially useful as a concrete example of how the different applications connect: the user generates a QR from the app, a branch processes it, and the backend validates and records the redemption." data-locale-es="Otra parte interesante del proyecto fue el propio flujo de fidelización. La plataforma soporta acumulación de puntos, gestión de productos y promociones, operaciones específicas por sucursal, historial de transacciones y canje mediante QR. Esta última funcionalidad fue especialmente útil como ejemplo concreto de cómo se conectan las distintas aplicaciones: el usuario genera un QR desde la app, una sucursal lo procesa y el backend valida y registra el canje.">Another interesting part of the project was the loyalty flow itself. The platform supports points accumulation, product and promotion management, branch-specific operations, transaction history, and QR-based redemption. That last feature was especially useful as a concrete example of how the different applications connect: the user generates a QR from the app, a branch processes it, and the backend validates and records the redemption.</p>

<p data-locale-en="The project also required us to think about much more than endpoints. We had to work on environment configuration, CORS, JWT authentication, database migrations, documentation for teammates, and development setups that simulated tenant subdomains locally. In practice, a big part of the work was making the system understandable and usable by the whole team." data-locale-es="El proyecto también nos obligó a pensar en mucho más que endpoints. Tuvimos que trabajar en configuración de entornos, CORS, autenticación JWT, migraciones de base de datos, documentación para compañeros y entornos de desarrollo que simularan subdominios de tenant en local. En la práctica, una parte grande del trabajo consistió en hacer que el sistema fuera entendible y utilizable para todo el equipo.">The project also required us to think about much more than endpoints. We had to work on environment configuration, CORS, JWT authentication, database migrations, documentation for teammates, and development setups that simulated tenant subdomains locally. In practice, a big part of the work was making the system understandable and usable by the whole team.</p>

<p data-locale-en="Because this was a group project, collaboration became part of the technical challenge as well. We documented architecture decisions, usage flows, and setup instructions to reduce friction between the backend, frontend, and mobile parts of the system. Looking back, that documentation was not a secondary task. It was necessary to keep the project moving." data-locale-es="Como este fue un proyecto grupal, la colaboración también se volvió parte del desafío técnico. Documentamos decisiones de arquitectura, flujos de uso e instrucciones de puesta a punto para reducir fricción entre backend, frontend y mobile. Mirando hacia atrás, esa documentación no fue una tarea secundaria: fue necesaria para que el proyecto siguiera avanzando.">Because this was a group project, collaboration became part of the technical challenge as well. We documented architecture decisions, usage flows, and setup instructions to reduce friction between the backend, frontend, and mobile parts of the system. Looking back, that documentation was not a secondary task. It was necessary to keep the project moving.</p>

<h2 data-locale-en="Conclusion" data-locale-es="Conclusion">Conclusion</h2>

<p data-locale-en="ServiPuntosUy was one of those projects where the most valuable learning did not come from a single feature, but from having to connect many moving parts into one coherent platform." data-locale-es="ServiPuntosUy fue uno de esos proyectos donde el aprendizaje mas valioso no vino de una funcionalidad aislada, sino de tener que conectar muchas piezas en una unica plataforma coherente.">ServiPuntosUy was one of those projects where the most valuable learning did not come from a single feature, but from having to connect many moving parts into one coherent platform.</p>

<p data-locale-en="Working on a multi-tenant system in .NET, while also coordinating with web and mobile clients, forced us to think more carefully about architecture, boundaries, and maintainability. It was a demanding academic project, but it also felt close to the kind of complexity that appears in real software products." data-locale-es="Trabajar en un sistema multi-tenant en .NET, mientras coordinábamos clientes web y mobile, nos obligó a pensar con más cuidado en arquitectura, límites y mantenibilidad. Fue un proyecto académico exigente, pero también se sintió cercano al tipo de complejidad que aparece en productos de software reales.">Working on a multi-tenant system in .NET, while also coordinating with web and mobile clients, forced us to think more carefully about architecture, boundaries, and maintainability. It was a demanding academic project, but it also felt close to the kind of complexity that appears in real software products.</p>
