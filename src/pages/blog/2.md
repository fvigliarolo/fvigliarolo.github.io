<p data-locale-en="When we started Alum-net, I did not see testing as a final validation step. I saw it as a design decision. The project had two constraints that changed everything: we were going to work through incremental deliveries, and the final system had to support both a web client and a mobile client." data-locale-es="Cuando empezamos Alum-net, yo no veía el testing como una validación final. Lo veía como una decisión de diseño. El proyecto tenía dos condiciones que cambiaban bastante el enfoque: íbamos a trabajar con entregas incrementales y el sistema final debía sostener tanto un cliente web como uno mobile.">When we started Alum-net, I did not see testing as a final validation step. I saw it as a design decision. The project had two constraints that changed everything: we were going to work through incremental deliveries, and the final system had to support both a web client and a mobile client.</p>

<p data-locale-en="That led me to a practical question very early on: where did it make the most sense to invest testing effort so the project could keep growing without breaking something important in every iteration?" data-locale-es="Eso me llevó bastante temprano a una pregunta práctica: dónde convenía poner el esfuerzo de testing para que el proyecto pudiera seguir creciendo sin romper algo importante en cada iteración.">That led me to a practical question very early on: where did it make the most sense to invest testing effort so the project could keep growing without breaking something important in every iteration?</p>

<h2 data-locale-en="No everything automated is better" data-locale-es="No todo lo automatizado es mejor">No everything automated is better</h2>

<p data-locale-en="One of the first decisions I made was not to push automation equally across every layer of the system. We were going to have web, mobile, and backend, but those layers were not in the same state. The UI was still going to be built in a fairly linear way, while the backend was already becoming the shared foundation for both clients." data-locale-es="Una de las primeras decisiones que tomé fue no empujar la automatización de la misma forma en todas las capas del sistema. Íbamos a tener web, mobile y backend, pero esas capas no estaban en la misma situación. La interfaz todavía se iba a construir de forma bastante lineal, mientras que el backend ya se estaba convirtiendo en la base compartida para ambos clientes.">One of the first decisions I made was not to push automation equally across every layer of the system. We were going to have web, mobile, and backend, but those layers were not in the same state. The UI was still going to be built in a fairly linear way, while the backend was already becoming the shared foundation for both clients.</p>

<p data-locale-en="That is why I did not want to automate the UI from the beginning. We still validated both web and mobile through formal end-to-end manual tests, but I did not think it was a good investment to automate a layer that still depended completely on the backend to make sense. In this case, automating everything early would have spread the effort too thin instead of improving confidence." data-locale-es="Por eso no quise automatizar la interfaz desde el principio. Igual validamos tanto web como mobile mediante pruebas end-to-end manuales formales, pero no me parecía una buena inversión automatizar una capa que todavía dependía por completo del backend para tener sentido. En este caso, automatizar todo demasiado temprano iba a dispersar el esfuerzo en lugar de aumentar la confianza sobre el sistema.">That is why I did not want to automate the UI from the beginning. We still validated both web and mobile through formal end-to-end manual tests, but I did not think it was a good investment to automate a layer that still depended completely on the backend to make sense. In this case, automating everything early would have spread the effort too thin instead of improving confidence.</p>

<h2 data-locale-en="Secure the core of the system first" data-locale-es="Asegurar primero el nucleo del sistema">Secure the core of the system first</h2>

<p data-locale-en="For me, the backend was the real core of Alum-net. It concentrated authentication, roles, courses, enrollments, sections, tasks, forums, grades, file handling, and several integrations. It also had to serve both web and mobile consistently. If that layer was unstable, the rest of the product would always be fragile." data-locale-es="Para mi, el backend era el verdadero nucleo de Alum-net. Concentraba autenticacion, roles, cursos, matriculas, secciones, tareas, foros, calificaciones, manejo de archivos y varias integraciones. Ademas, tenia que servir de forma consistente tanto a web como a mobile. Si esa capa era inestable, el resto del producto siempre iba a ser fragil.">For me, the backend was the real core of Alum-net. It concentrated authentication, roles, courses, enrollments, sections, tasks, forums, grades, file handling, and several integrations. It also had to serve both web and mobile consistently. If that layer was unstable, the rest of the product would always be fragile.</p>

<p data-locale-en="Because of that, I approached testing as a way to preserve backend sanity while the system kept evolving. The goal was not only to prove that a new feature worked. The real goal was to prevent regressions and make sure new behavior could be added without silently damaging what we already had." data-locale-es="Por eso encare el testing como una forma de preservar la sanidad del backend mientras el sistema seguia evolucionando. El objetivo no era solamente demostrar que una funcionalidad nueva andaba. El objetivo real era prevenir regresiones y asegurarnos de que el comportamiento nuevo pudiera incorporarse sin danar en silencio lo que ya teniamos.">Because of that, I approached testing as a way to preserve backend sanity while the system kept evolving. The goal was not only to prove that a new feature worked. The real goal was to prevent regressions and make sure new behavior could be added without silently damaging what we already had.</p>

<h2 data-locale-en="Prioritize before covering" data-locale-es="Priorizar antes de cubrir">Prioritize before covering</h2>

<p data-locale-en="Once the backend had a minimal working base, I started prioritizing which endpoints had to be verified first. That prioritization followed the same logic the developers were already using for implementation: critical features came first. Instead of chasing broad coverage from the start, I wanted the most important parts of the system to be protected first." data-locale-es="Una vez que el backend tuvo una base minima funcional, empece a priorizar que endpoints habia que verificar primero. Esa priorizacion siguio la misma logica que ya estaban usando los desarrolladores para implementar: primero iban las funcionalidades mas criticas. En lugar de perseguir cobertura amplia desde el comienzo, queria que las partes mas importantes del sistema quedaran protegidas primero.">Once the backend had a minimal working base, I started prioritizing which endpoints had to be verified first. That prioritization followed the same logic the developers were already using for implementation: critical features came first. Instead of chasing broad coverage from the start, I wanted the most important parts of the system to be protected first.</p>

<p data-locale-en="That meant focusing first on areas such as authentication, access by role, users, courses, content, enrollments, and other flows that had a strong impact on the rest of the system. For this project, deciding what to protect first was more useful than trying to cover everything evenly." data-locale-es="Eso significo enfocarme primero en areas como autenticacion, acceso por rol, usuarios, cursos, contenidos, matriculas y otros flujos que tenian mucho impacto sobre el resto del sistema. En este proyecto, decidir que proteger primero fue mucho mas util que intentar cubrir todo de forma pareja.">That meant focusing first on areas such as authentication, access by role, users, courses, content, enrollments, and other flows that had a strong impact on the rest of the system. For this project, deciding what to protect first was more useful than trying to cover everything evenly.</p>

<h2 data-locale-en="Turning the idea into a workflow" data-locale-es="Convertir la idea en un flujo de trabajo">Turning the idea into a workflow</h2>

<p data-locale-en="The strategy only made sense if it became part of the team's daily process. Together with my testing teammate, I worked on the test plan before implementing each batch of endpoint tests. Then we built the cases in Postman, including pre-request and post-request logic where needed, and kept adding them to the collection that would later run automatically." data-locale-es="La estrategia solo tenia sentido si se convertia en parte del proceso diario del equipo. Junto con mi companero de testing, trabaje el plan de pruebas antes de implementar cada tanda de tests sobre endpoints. Despues armamos los casos en Postman, incluyendo logica de pre-request y post-request cuando hacia falta, y los fuimos agregando a la coleccion que luego iba a correr automaticamente.">The strategy only made sense if it became part of the team's daily process. Together with my testing teammate, I worked on the test plan before implementing each batch of endpoint tests. Then we built the cases in Postman, including pre-request and post-request logic where needed, and kept adding them to the collection that would later run automatically.</p>

<p data-locale-en="What we wanted to verify was straightforward: that each endpoint returned the structure the frontend would need, that status codes were consistent and meaningful, and that new functionality did not break previous behavior. Postman gave us a practical way to design and maintain those checks, and Newman let us run the same collection without the UI." data-locale-es="Lo que queriamos verificar era bastante directo: que cada endpoint devolviera la estructura que luego iba a necesitar el frontend, que los status codes fueran consistentes y tuvieran criterio, y que una funcionalidad nueva no rompiera el comportamiento previo. Postman nos dio una forma practica de disenar y mantener esas verificaciones, y Newman nos permitio ejecutar la misma coleccion sin depender de la interfaz.">What we wanted to verify was straightforward: that each endpoint returned the structure the frontend would need, that status codes were consistent and meaningful, and that new functionality did not break previous behavior. Postman gave us a practical way to design and maintain those checks, and Newman let us run the same collection without the UI.</p>

<p data-locale-en="The final step was integrating that collection into GitHub Actions, so the test suite would run on every pull request to main. That turned the tests into a real quality gate. A feature was not ready to be merged just because it worked locally. Before reaching the main branch, it had to pass the automated checks that protected the backend from regressions." data-locale-es="El paso final fue integrar esa coleccion en GitHub Actions para que la bateria corriera en cada pull request hacia main. Eso convirtio a los tests en una barrera real de calidad. Una funcionalidad no estaba lista para integrarse solo porque anduviera en local. Antes de llegar a la rama principal, tenia que pasar por las verificaciones automaticas que protegian al backend de regresiones.">The final step was integrating that collection into GitHub Actions, so the test suite would run on every pull request to main. That turned the tests into a real quality gate. A feature was not ready to be merged just because it worked locally. Before reaching the main branch, it had to pass the automated checks that protected the backend from regressions.</p>

<h2 data-locale-en="What manual testing still solved" data-locale-es="Lo que las pruebas manuales seguian resolviendo">What manual testing still solved</h2>

<p data-locale-en="Manual testing still had an important role. It was the right place to validate the final behavior of web and mobile, and it also helped us detect inconsistencies that were not necessarily breaking the system but still mattered for the product. In the manual testing records we completed 49 tests, with no critical failures, although some cases were marked as improvable because they revealed validation gaps or messages that were too technical for end users." data-locale-es="Las pruebas manuales siguieron teniendo un rol importante. Eran el lugar correcto para validar el comportamiento final de web y mobile, y tambien nos ayudaban a detectar inconsistencias que no necesariamente rompian el sistema, pero que igual importaban para el producto. En el registro de pruebas manuales completamos 49 casos, sin fallas criticas, aunque algunos quedaron marcados como mejorables porque mostraban huecos de validacion o mensajes demasiado tecnicos para el usuario final.">Manual testing still had an important role. It was the right place to validate the final behavior of web and mobile, and it also helped us detect inconsistencies that were not necessarily breaking the system but still mattered for the product. In the manual testing records we completed 49 tests, with no critical failures, although some cases were marked as improvable because they revealed validation gaps or messages that were too technical for end users.</p>

<h2 data-locale-en="What this approach solved, and what it did not" data-locale-es="Lo que este enfoque resolvio, y lo que no">What this approach solved, and what it did not</h2>

<p data-locale-en="This strategy did not eliminate every problem. During development, we still had friction when backend changes modified API responses in ways that affected frontend expectations or existing tests. But that is exactly why I still think the decision was right. In an incremental project, the main risk is not only a new feature failing. The bigger risk is new behavior silently damaging old behavior. Our testing approach gave us a much clearer way to catch those issues before they reached main." data-locale-es="Esta estrategia no elimino todos los problemas. Durante el desarrollo igual tuvimos fricciones cuando cambios en el backend modificaban respuestas de la API de una forma que afectaba las expectativas del frontend o los tests existentes. Pero justamente por eso sigo pensando que la decision fue correcta. En un proyecto incremental, el riesgo principal no es solo que falle una funcionalidad nueva. El riesgo mayor es que un comportamiento nuevo dane en silencio un comportamiento anterior. Nuestro enfoque de testing nos dio una forma mucho mas clara de detectar eso antes de llegar a main.">This strategy did not eliminate every problem. During development, we still had friction when backend changes modified API responses in ways that affected frontend expectations or existing tests. But that is exactly why I still think the decision was right. In an incremental project, the main risk is not only a new feature failing. The bigger risk is new behavior silently damaging old behavior. Our testing approach gave us a much clearer way to catch those issues before they reached main.</p>

<h2 data-locale-en="Conclusion" data-locale-es="Conclusion">Conclusion</h2>

<p data-locale-en="Looking back, the most important lesson was that testing should not always be distributed evenly across every layer. In Alum-net, the best decision was to stabilize the backend first, reserve end-to-end validation of web and mobile for formal manual checks, and turn automated API tests into part of the integration flow. It was not about testing more for the sake of it. It was about making sure the project could keep growing without constantly undermining its own foundation." data-locale-es="Mirando hacia atras, la leccion mas importante fue que el testing no siempre conviene distribuirlo de forma pareja entre todas las capas. En Alum-net, la mejor decision fue estabilizar primero el backend, reservar la validacion end-to-end de web y mobile para instancias manuales formales, y convertir los tests automaticos de API en parte del flujo de integracion. No se trataba de testear mas por testear. Se trataba de asegurarnos de que el proyecto pudiera seguir creciendo sin socavar constantemente su propia base.">Looking back, the most important lesson was that testing should not always be distributed evenly across every layer. In Alum-net, the best decision was to stabilize the backend first, reserve end-to-end validation of web and mobile for formal manual checks, and turn automated API tests into part of the integration flow. It was not about testing more for the sake of it. It was about making sure the project could keep growing without constantly undermining its own foundation.</p>
